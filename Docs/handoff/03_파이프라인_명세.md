## §0. 공용 데이터 객체

```python
PipelineState = {
  "request_id": str,
  "raw_question": str,                  # 불변
  "corrected_question": str,
  "corrections": [Correction],          # {"from","to","source":"L1|L2|L3|L4"}
  "goal_type_surface": str|None,        # 어미 판독 타입 (검산 전용)
  "hops": [Hop],
  "final_op": FinalOp|None,
  "hop_results": {int: HopResult},
  "mode": "FULL"|"SIMPLE",
  "answer": Answer|None,
  "trace": [TraceEvent],
}

Hop = {
  "id": int,
  "start": str,                         # 리터럴 or "{n}" 참조
  "start_node": str|None,               # 그라운딩 후 노드 ID
  "relation": str,                      # 스키마 10종 + "relatedTo"
  "relation_grounded": str|None,
  "direction": "fwd"|"inv",
  "expected": {
     "type": str,                       # PERSON|LOCATION|ORG|DATE|NUMBER|WORK|TERM
     "definition": str,                 # 조건의 재서술. 정답 고유명사 금지 (검증 8번)
  },
  "anchor_meta": {"name": str, "disambiguator": str|None},
  "sub_query": str,
  "hint_relations": [str]|None,         # 그라운딩 포기 시 실존 relation 목록
}

FinalOp = {"operator": "EARLIER|LATER|SAME|GREATER|SMALLER", "in": [int]}
# 연산자 요구 타입: EARLIER/LATER→DATE, GREATER/SMALLER→NUMBER, SAME→any
# yes/no 출력 연산의 답 포맷은 "yes"/"no" 고정

Candidate = {
  "entity": str,
  "origin": "vector"|"graph"|"both",
  "evidence_paragraph_ids": [str],
  "verification": Verification|None,
  "score": float,
}

Verification = {
  "type_check": bool|None,              # None = 약화모드 미시행
  "name_check": bool|None,
  "desc_check": bool,
  "backlink_check": "pass"|"absent"|"contradict"|None,   # 3값 (C1). None=약화모드 미시행
  "passed": bool,                       # 산식 아래 (C1)
  "weak_mode": bool,
}
# passed 규칙 (C1 — backlink 단독 실패가 pass_ratio에 묻히는 문제 해소):
#  1. backlink_check == "contradict" → passed=False (hard-veto, pass_ratio 무관)
#     "contradict" = 그래프에 동일 원본 relation의 '다른' tail이 실존 → 이 후보는 그 relation의 답이 아님
#  2. 그 외: backlink "absent"는 중립(시행수에서 제외), 나머지 시행 검사 통과율 ≥ pass_ratio
#  → 시나리오 B: '빈'은 모차르트-bornIn→잘츠부르크가 그래프에 실존하므로 "contradict" = veto (재질의로 회복)

HopResult = {
  "hop_id": int, "answer": str,
  "candidates": [Candidate],            # ★ 전량 보존 (oracle 분석 원천)
  "tie": bool, "branched": bool, "retries": int,
  "status": "OK"|"WEAK"|"FAILED",
  "queries_used": [str],
}

Answer = {
  "text": str,
  "status": "ANSWERED"|"PARTIAL"|"UNVERIFIABLE",
  "confidence": float,                  # 시행 검사 통과율의 hop 평균 + path_check 가산
  "path_check": bool|None,
  "evidence": [{"paragraph_id": str, "hop_id": int}],
  "verified_until": int|None,
}
```

## §1. 단계별 계약 (읽는 필드와 쓰는 필드)

### [0] 교정 L1~L3 (결정적, LLM 금지)
- 읽기는 raw_question이고, alias 사전과 약어 사전에 의존한다.
- L1 규칙: NFC 정규화, 공백, 중복 조사, 약어 확장.
- L2 사전 대조: 핵심 명사구가 전부 alias에 있으면 오타가 없다고 판정하고, L3·L4를 건너뛰는 플래그를 세운다.
- L3 오타 교정은 미매칭 토큰에만 적용한다. 후보는 alias 키 중 편집거리≤2인 것과 자모거리 후보의 합집합이다. **후보가 유일하고 거리≤1일 때만 확정 교정한다.** 애매하면 후보 목록을 [1]에 전달한다.
- 쓰기: corrected_question(잠정), corrections, 미해결 토큰과 후보.
- 실패는 없다. 못 고치면 그대로 넘긴다.

### [1] 분해 통합 콜 (LLM 1콜, structured output)
- 프리필터: 관계 표지가 0개이고 단문이면 콜을 건너뛰고 mode=SIMPLE로 둔다.
- 읽기: corrected_question(잠정), 미해결 토큰과 후보, relation 스키마(프롬프트 상수).
- 쓰기: corrected_question(확정), corrections(L4), goal_type_surface, hops[], final_op.
- 프롬프트 제약: relation은 스키마 안에서만 고른다. 교정은 제시한 후보 중 선택하거나 원문을 유지하되 diff를 강제한다. start 참조는 "{n}" 형식을 쓴다. definition은 조건 재서술만 담고 고유명사는 금지한다. fwd·inv 방향 few-shot 2개를 반드시 넣는다.
- 실패: 파싱에 실패하면 재생성을 1회 하고, 그래도 안 되면 SIMPLE로 간다.

### [2] 계획 검증 8종 (코드, LLM 0콜)
| # | 검사 | 실패 처리 |
|---|---|---|
| 1 | relation ∈ 스키마+relatedTo | 재생성 1회 → SIMPLE |
| 2 | direction 유효 | 〃 |
| 3 | 참조 무결성(실존·DAG·op.in), 위상정렬 1회 | 〃 |
| 4 | 깊이≤3 AND 총hop≤5 | 절단 or SIMPLE |
| 5 | 타입체인: leaf(또는 op출력) == goal_type_surface | 재생성 1회 |
| 6 | op 입력 타입 == 연산자 요구 타입 | final_op 제거 후 진행 |
| 7 | 병렬의 억지 직렬화 감지 | 참조 제거·분리 |
| 8 | definition 정답 누출: definition 내 그 hop의 답(tail)과 후속 hop 참조 엔티티명 검출(앵커·조건 서술의 기존 엔티티는 허용. 예: "베토벤이 태어난 도시"의 '베토벤'은 허용, C5) | 재생성 1회 |
- 쓰기: execution_order(위상정렬 결과).

### [3] 그라운딩 (LLM 0~1콜)
- anchor: alias exact 매칭을 먼저 보고, 복수면 disambiguator와 노드 설명 유사도로 가른 뒤, 그래도 안 되면 임베딩 top-1을 쓴다. 끝내 못 풀면 graph_miss로 마킹하고 진행한다.
- relation: 대부분 exact로 맞춘다. 안 되면 cos>θ_high일 때 스냅하고, θ_low~θ_high 구간이면 객관식 1콜로 타이브레이크한다. cos<θ_low이면 포기하고 relation_grounded=None으로 두되 hint_relations를 첨부한다.
- 쓰기: start_node, relation_grounded, hint_relations.

### [실행기] ready-set 루프
```python
async def execute(state):
    while 미완료 hop 존재:
        ready = [h for h in hops if h.id not in done and start_해석가능(h, done)]
        results = await gather(*[resolve_hop(h, state) for h in ready])
        done.update(results)
    # tie 발생 시: 후속 hop을 chain_id별 복제, 활성 체인 ≤3
```
- "{n}" 치환은 실행 시점에 한다. 재귀 함수 구현은 금지한다 (ADR-3).

#### [4] 검색 (LLM 1콜)
- 경로A: FAISS(ADR-11: Chroma→FAISS)에 sub_query(definition을 결합한 임베딩, HyDE)로 top-8 문단을 뽑고, LLM 1콜로 후보를 추출한다.
- 경로B: graph.neighbors(start_node, relation_grounded, direction)로 노드와 근거 문단ID를 얻는다.
- 병합: alias 기준으로 같은 엔티티면 origin="both"로 묶는다. graph 출신은 backlink="pass"로 기록해 자동 통과시키되, **"contradict" 검사는 면제하지 않는다.** 스냅으로 오염된 후보가 veto를 우회하는 구멍을 막기 위해서다(C1/ADR-6의 2겹 그물 유지).
- 후보가 0이면 즉시 [6]으로 간다.

#### [5] 검증·확정 (LLM 0콜)
- type: node.type == expected.type인지 본다(direction 반영).
- name: 후보가 alias에 있는지 본다.
- backlink: inv는 inverse(분해 의도 원본 relation)이다(grounded가 아님, ADR-6). 3값으로 판정한다(C1).
    - has_edge(후보→start_node, inv)이면 "pass".
    - 엣지가 없고, 후보가 아닌 t'로 has_edge(t'→start_node, inv)가 있으면 "contradict"(hard-veto).
    - 엣지가 없고 그런 t'도 없으면 "absent"(중립, 시행수 제외).
- desc: cos(후보 근거문단 또는 노드 설명, expected.definition) > θ_desc인지 본다.
- graph_miss면 weak_mode로 두고(type/name/backlink 중 생략 가능한 것만 시행).
- score = 통과수 + λ·벡터유사도. top1이 passed면 확정한다. margin<ε이면 tie. shortcut으로, 문단에서 최종 leaf 타입의 답을 동시에 검출하면 잔여 hop을 건너뛰는 플래그를 세운다.
- 전원 미달이면 [6]으로 간다.

#### [6] 재질의 (LLM 1콜, hop당 max 2)
- 프롬프트에는 검사별 실패 사유, queries_used(중복 금지), hint_relations 또는 이웃 relation 목록을 주입한다.
- 소진하면 status=FAILED로 두고 체인을 중단한다.

### [7] final_op (코드)
- 날짜와 숫자를 정규화한 뒤 연산한다. 파싱에 실패하면 PARTIAL로 강등하고 두 값을 나열한다.

### [8] Path-Check + 체인 중재 (코드)
- path_exists(anchor₀, [의도 relation들], answer)를 제약 BFS로 확인한다(RoG Algorithm 1 방식).
- 경로가 존재하면 confidence를 가산하고 뱃지를 붙인다. 부재하면 중립으로 두고 감점하지 않는다.
- 체인이 복수면 검증 점수 합과 path_check로 승자를 정한다.

### [9] 응답 조립 (LLM 1콜)
- ANSWERED / PARTIAL / UNVERIFIABLE 중 하나로 낸다. UNVERIFIABLE은 "hopN까지 확인: {중간결과}, 부족: {사유}" 형식과 verified_until을 함께 담는다.

### [로깅]
- 모든 단계의 진입과 이탈을 TraceEvent로 남긴다. 필수 이벤트는 correction_applied, decompose_ok/fallback, plan_validation_fail(#), grounding_snap/tiebreak/miss, candidates_merged, verify_result, tie_detected, requery(사유), shortcut, path_check, answer_status이다.
- candidates는 전량 보존한다 (ADR-8).

## §2. 모드 매트릭스
| mode | 경로 | 용도 |
|---|---|---|
| FULL (Ours) | 전 단계 (verifier=quad) | 본 시스템 |
| SIMPLE | [0]→경로A 1회→[9] | 분해가 불가능하거나 불필요한 질문의 폴백 |
| Agent-basic | [0]→원질의(분해 없음) 경로A→llm_judge→재검색→[9] | 현실 대조군, 타 팀 공통선(분해 미사용이라 verifier 주입 구조가 아님, C4) |
| Ours−G (ablation) | FULL과 동일, verifier=llm_judge만 교체(verifier 주입) | 이득이 분해가 아니라 검증 방식에서 온다는 것을 분리 증명. 분해 사용 (ADR-4, C4) |
| Baseline | 원질의→경로A 1회→[9] | 기준선 |

## §3. LLM 콜 예산
2-hop이 정상이면 4콜이다. 분기와 재시도가 겹치면 상한은 약 8콜이다. 트리플 추출은 인덱싱 시 1회성이며 문단 상한으로 통제한다.
