# 07. 첫 번째 태스크 — W0.1 KorQuAD 수율 체크 (지금 바로)

## 왜 이게 1번인가
현재 데이터 전략에는 **아직 검증하지 못한 가정이 2개** 있는데, 이 스크립트 하나로 둘 다 확인할 수 있다.

1. "KorQuAD에서 브릿지 가능 쌍(answer(QA1)==entity(QA2))이 충분히 나온다." 안 나오면 멀티홉 평가셋 전략이 무너진다.
2. "코퍼스를 평가 문항 원문서+이웃으로 역정의하면 정합이 보장된다." 코퍼스 문서 목록이 여기서 나온다.

부산물로 relation 분포가 나오는데, 이걸로 relations.yaml(W0.2)을 가설이 아니라 실측으로 확정한다.

## 스펙: scripts/w0_yield_check.py

### 입력
- KorQuAD 1.0 train+dev JSON. https://korquad.github.io/ 에서 받아 data/raw/에 둔다.
- (선택) 한국어 위키 redirect 목록. 없으면 문자열 매칭만으로 1차 수행한다.

### 처리
1. 전 QA를 로드해 (qid, question, answer, 문서제목, 문단) 형태로 평탄화한다.
2. 정답 엔티티를 정규화한다. answer에서 조사를 떼고, 괄호 보조표기를 분리한다 (예: "본(Bonn)" → {본, Bonn}).
3. 브릿지를 매칭한다. answer(QA_a)가 question(QA_b)에 부분 문자열로 등장하는 (a,b) 쌍을 찾는다.
   - a와 b의 원문 문서가 서로 달라야 한다 (같은 문서면 멀티홉이 아니다).
   - answer가 2글자 이하이거나 일반명사(불용어 목록)면 제외한다 (오매칭 방지).
   - 매칭 방식은 exact / josa-stripped / alias로 태깅한다.
4. relation을 휴리스틱으로 태깅한다. QA_a와 QA_b 질문 각각에 06 문서 relations.yaml 초안의 ko 표현 패턴을 매칭해 등장 relation 분포를 집계한다. 미매칭은 "unknown"으로 집계하고, unknown 상위 표현도 함께 출력해 새 relation 후보를 발굴한다.
5. 코퍼스 문서 목록을 만든다. 브릿지 쌍과 단일홉 평가 후보 문항들의 원문 문서 제목 집합을 산출한다.

### 출력
- `data/w0/bridge_candidates.jsonl` — {qa_a, qa_b, bridge_entity, match_type, relation_a, relation_b}
- `data/w0/relation_stats.json` — relation별 빈도와 unknown 상위 표현 30개
- `data/w0/corpus_doc_list.txt` — 문서 제목 목록
- `data/w0/report.md` — 아래 판정 기준을 자동으로 채운다

### 완료 기준 (report.md에 명시)
| 항목 | 판정 |
|---|---|
| 브릿지 쌍 수 ≥ 300 | PASS면 진행. FAIL이면 KorQuAD 2.0 확장이나 위키 하이퍼링크 보조 여부를 결정해야 함 |
| relation 상위 10종 커버리지 (전체 쌍 중 상위 10 relation으로 태깅되는 비율) | ≥60%면 스키마 10종 유지, 미만이면 스키마 교체 |
| 코퍼스 문서 수 | 하이퍼링크 이웃 포함 예상 문단 수가 3만 이내인지 추정 |

### 구현 메모
- LLM은 쓰지 않는다 (전부 문자열 처리라 비용 0, 결과 결정적).
- 조사 제거는 간단한 규칙으로 시작한다 (이/가/은/는/을/를/의/에서/으로/로 어미 strip). kiwipiepy는 결과가 부실할 때만 도입한다.
- 실행 시간은 5분 미만을 목표로 하고, 진행률을 출력한다.
- 이 스크립트가 W1.6 build_multihop_set.py의 입력을 만든다. jsonl 필드를 누락하면 안 된다.

## 이후 순서 (스크립트 결과 확인 후)
1. report.md 판정을 확인하고 relations.yaml을 확정한다 (W0.2).
2. FAIL이면 사람에게 보고한 뒤 대안을 결정한다 (임의 진행 금지).
3. PASS면 04_WBS.md의 W1부터 진행한다.
