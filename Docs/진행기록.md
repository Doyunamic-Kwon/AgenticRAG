# 진행 기록 (실행 로그)

`handoff/`가 설계 계약(불변 스펙)이라면, 이 문서는 **실제로 무엇을 돌렸고 어떤 수치가 나왔는지**의 시간순 기록이다. 결정의 근거는 `handoff/02`(ADR)에, 수치·여정은 여기에 남긴다.

기간: W0~W1 (2026-07-22 ~ 07-23). 제공자: Upstage solar (임베딩 4096d + solar-pro), OpenAI 후순위.

## W0.1 — KorQuAD 수율 체크
- 입력: KorQuAD 1.0 train+dev 66,181 QA. LLM 0콜, 문자열 처리 5초.
- 원시 브릿지 234,186쌍 / "빌드가능(양쪽 hop 스키마 relation)" 1,969쌍 / 코퍼스 1,545문서·10,621문단.
- 판정: 가정 2개(브릿지 충분·코퍼스 역정의) 확인. **단, "1,969 빌드가능"은 ko-문자열 매칭 태그가 만든 낙관치**였음(W1.6에서 드러남).

## W1.1 — 코퍼스 구축
- corpus_doc_list의 KorQuAD 문서를 문단 단위로 → `data/corpus.jsonl` 10,615문단 / 1,545문서 / 평균 518자.
- 제한: 위키 덤프 없이 KorQuAD 문단만 → backlink 이웃은 이 범위로 한정(W1.3에서 반영).

## W1.2 — FAISS 인덱싱 + 검색 스팟체크
- solar-embedding-1-large(dim 4096) → FAISS flat(IndexFlatIP, 코사인). 입력 5000자 절단(solar 4000토큰 한도, 실측).
- **단일홉 검색 품질 (n=5,000): Recall@1 83.2% · @5 97.2% · @10 98.5%.**
- 해석: 검색 스택 건강. 단일홉은 질문이 원문단 파생이라 쉬운 케이스.
- 결정 근거: `handoff/02` ADR-11 (벡터백엔드·임베딩·검색단위·batch).

## W1.5 — 평가 하네스 + Baseline
- `eval/metrics.py`(Recall@k·MRR·nDCG 순수함수) + `eval/run_eval.py --mode baseline`.
- **Baseline 측정 (Day 2 게이트):**

  | 셋 | Recall@5 | Recall@10 | MRR | nDCG@10 |
  |---|---|---|---|---|
  | 단일홉 (n=300) | 97.7% | 98.7% | 0.898 | 0.920 |
  | 멀티홉 (2Wiki, n=26) | 50.0% | 61.5% | 0.319 | 0.391 |

- 해석: 멀티홉에서 Baseline이 **37p 낮음** → "복합질문 1회 검색은 무너진다"는 문제의 정량 증명. (2Wiki 셋에 shortcut 필터 미적용이라 61.5%; 적용 시 더 낮아짐.)
- basic/ours_g/ours 모드는 파이프라인 완성 후(W4).

## W1.6 — 멀티홉 평가셋: KorQuAD 실패 → 2Wiki 전환
### 시도 A: KorQuAD 브릿지 자동 생성 (폐기)
- 브릿지 → LLM 패러프레이즈 → shortcut 필터로 gold 생성 시도.
- 결과: solar-pro·gpt-4o-mini **두 심판 모두 clean 사슬 ~3% 수율에 그마저 깨짐**(사람을 memberOf 대상으로, 날짜를 bornIn 답으로 등).
- 진단: KorQuAD 답↔질문 브릿지는 **어휘적 우연**이지 관계 사슬이 아님. 소스 한계(모델 문제 아님). W0.1 "1,969"는 허수.

### 조사: 기존 한국어 멀티홉 셋
- 없음. KoBLEX=법률, KorQuAD2=문서내 다중문단, AI Hub=뉴스 MRC → 부적합. 깨끗한 2-hop(2Wiki·HotpotQA)은 전부 영어.

### 시도 C: 2WikiMultihopQA 한국어 적응 (채택)
- 2Wiki는 (entity, relation, entity) 트리플 + 추론유형 내장 → **관계·타입 gold가 데이터에 있음**(LLM 추측 불필요). 라이선스도 파생 허용.
- 파이프라인: compositional 필터 → 관계를 우리 스키마로 매핑 → 엔티티 en→ko langlinks 정합 → ko 위키 intro를 코퍼스 증분 → solar-pro 질문 한국어화.
- **프로토타입 9/10 clean** (KorQuAD 0/10 대비). 확대 결과 **26문항** (관계매핑 330/850=39% → 한국어정합 → 채택 26).
- 산출: `data/eval/multihop_2wiki.jsonl` 26문항 + `data/corpus_2wiki.jsonl` 52문단.
- **한계: createdBy→{bornIn 15·nationality 6·diedIn 4·memberOf 1} 편향** (전부 "창작자→인물 속성" 단일 패턴). 다양성 위해 A(KorQuAD 정제, 병렬 진행)와 합산 필요.

### 통합
- `corpus.jsonl` + `corpus_2wiki.jsonl` = 10,667문단 재인덱싱(증분 임베딩 52개, 기존 재사용).
- 이 인덱스에 멀티홉 Baseline 측정 → 위 표.
