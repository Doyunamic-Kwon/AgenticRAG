# 01. 기획서 (개발 기준판)

## 1. 과제 요구사항 (외부 제약, 변경 불가)

프로젝트 12번 브리프가 요구하는 루프는 이렇다. 질의를 계획하고, 검색하고, 검색된 문서의 관련성을 평가한 뒤, 부족하면 질의를 교정·재정규화해서 다시 검색한다. 데모에서는 세운 계획, 재작성한 질의, 반복 횟수, 선택한 근거, 출처를 화면에 드러내야 한다. 권장 스택은 Python, ChromaDB, text-embedding-3-small, gpt-4o-mini, LangGraph(또는 직접 짠 상태 루프), Streamlit이다.

| ID | 브리프 요구 | 구현 |
|---|---|---|
| R1 | 질의 계획 | 질문 분해(hop 그래프) + ready-set 실행 계획 |
| R2 | 검색 | Chroma 벡터 + 그래프 이웃 이중 경로 |
| R3 | 검토 | Quad Verification (차별화 지점) |
| R4 | 교정·재정규화 | 4계층 교정 + 그래프 힌트 재질의 |
| R5 | 재검색 루프 | hop당 최대 2회 + "확인 불가" 탈출 |
| R6 | 과정 노출 데모 | Streamlit hop trace + 검증 뱃지 |
| R7 | 검색 성능 측정 | 3단 비교 (Recall@k / MRR / nDCG) |

## 2. 해결하는 문제

복합 질문을 한 번에 검색하면 hop마다 필요한 문서가 뒤섞여 정밀도가 무너진다. "모차르트가 태어난 도시의 대표 축제"를 통째로 던지면 모차르트 문서와 축제 문서가 같은 순위 경쟁을 하게 된다.

더 심각한 건 중간 오류가 검증 없이 전파되는 경우다. 첫 hop에서 잘못 고른 답이 그대로 다음 단계로 넘어가면, 시스템은 자신 있게 틀린 최종 답을 내놓는다. 우리가 가장 신경 쓰는 지점이 여기다.

마지막으로 검토 과정이 불투명하다. LLM이 통과와 탈락을 판정하면 왜 그렇게 판정했는지 근거가 남지 않는다.

## 3. 핵심 가치

틀린 중간 결과를 답이 되기 전에 걸러내고, 걸러낸 이유를 보여준다. 속도나 UI보다, 검증에 실패한 뒤 재질의로 회복하는 과정을 정량적·시각적으로 증명하는 것을 앞에 둔다.

## 4. Agent 자율성 범위

에이전트가 스스로 정하는 것은 분해 방식, 검색 경로, 후보의 채택과 기각, 재질의 여부와 내용, 그리고 지름길이 보일 때 남은 hop을 건너뛸지다.

반대로 멈춰야 하는 지점도 분명하다. 재질의 2회를 다 쓰면 추측하지 않고 "확인 불가"로 끝낸다. 분해 검증에 실패하면 SIMPLE로 내려간다. 주관적 비교는 거부한다.

원칙은 하나로 정리된다. 자율성의 상한은 검증할 수 있는 범위까지다.

## 5. 데이터 전략

코퍼스는 한국어 위키를 쓰되, 인물 카테고리를 크롤링하는 방식이 아니라 평가 문항의 원문 문서와 그 하이퍼링크 이웃으로 역으로 정의한다. 정답 문서가 코퍼스에 없으면 Recall 측정 자체가 무의미해지기 때문에, 코퍼스와 평가셋의 정합을 이 방식으로 보장한다.

단일홉 평가는 KorQuAD를 해당 문서 범위로 필터해서 쓴다. 멀티홉 평가는 100~200문항을 자체 구축하는데, KorQuAD QA 쌍을 브릿지로 잇는 방식이다(한 QA의 답이 다른 QA의 엔티티가 되는 쌍). 문항마다 중간 정답, 근거 문단, relation 사슬을 남겨야 한다(05 참조).

주의할 점이 둘 있다. KorQuAD 1.0은 CC BY-ND라 변형 데이터를 재배포할 수 없으므로, 재생성 스크립트만 공개한다. 또 평가셋을 시스템이 만든 그래프 경로에서 역생성하면 순환 평가가 된다. 평가셋의 원천인 KorQuAD는 시스템이 추출한 그래프와 독립이어야 한다.

## 6. KPI

- 1차 기준: 멀티홉 최종 Recall@10이 Agent-basic 대비 +X%p(측정 후 확정)에 Ours 절대치를 함께 본다. Baseline 대비 수치는 shortcut 필터 구성상 자동으로 달성되므로 헤드라인으로 쓰지 않는다(05 §4 참조). 여기에 Ours가 Ours−G보다 나은지로 검증의 기여를 분리해 증명한다.
- 보조: 중간 정답률, 오류 전파 차단율, 재질의 성공 전환율, 질문당 콜 수·비용·latency.

## 7. MVP 경계

반드시 되어야 하는 것은 R1~R7 전부, 직렬(CHAIN) 완전 지원, 시나리오 A/B/C 재현, 3단 비교 수치, 로컬 실행이다.

시간이 남으면 STAR(팬아웃), 결정적 비교(final_op), 나란히 비교 UI를 붙인다.

이번 범위에서 뺀 것은 주관적 비교(MORE_FAMOUS), FILTER/AGGREGATE/SUPERLATIVE, 4-hop 이상, 역질문 대화, 전면 빔서치, 대화 메모리, 배포와 동시성이다.

## 8. 데모 시나리오 (개발 완료 판정 기준이기도 하다)

- A (기본): "베토벤이 태어난 도시가 속한 나라는?" 분해를 보여주고, hop마다 검증 뱃지를 띄운 뒤, 근거와 함께 "독일"을 낸다.
- B (하이라이트): "모차르트가 태어난 도시의 대표 축제는?" 후보 '빈'이 backlink 검사에서 모순으로 걸려 탈락한다(그래프에 모차르트의 출생지가 잘츠부르크로 있기 때문). 이어 힌트를 넣어 재질의하고 '잘츠부르크'로 확정한 뒤 다음 hop으로 넘어간다. LLM judge였다면 '빈'을 통과시켰을 케이스라, 비교 모드로 그 차이를 함께 보여준다.
- C (오타): "베토밴 교향곡 9번 초연 장소" L3 교정이 붙고, SIMPLE로 판정해 바로 답한다.
