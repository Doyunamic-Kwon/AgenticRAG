# 05. 평가 명세

## 1. 3단 비교 (동일 질문셋, 동일 코퍼스)
| 모드 | 구성 | 존재 이유 |
|---|---|---|
| Baseline | 원질의 1회 벡터 검색 (분해·검증 없음) | 문제 존재 증명 — **KPI 대조군 아님** (§2 shortcut 필터의 판정자라 최종 Recall≈0, C2) |
| Agent-basic | 교정 + 원질의(분해 없음) LLM judge 검토 + 재검색 | 현실 대조군 "타 팀 공통선" = **주 KPI 비교선** |
| Ours−G (ablation) | Ours 전체 파이프라인 + verifier=llm_judge (verifier 주입) | 이득이 분해 아닌 그래프 검증임을 분리 증명 (거의 무료, ADR-4/C4) |
| Ours | + 분해 + 그라운딩 + Quad/Path 검증 + 힌트 재질의 | 차별화의 존재 증명 |
공정성 원칙: 네 모드가 검색 코드·인덱스·top_k를 공유. **Ours vs Agent-basic = 파이프라인 전체 효과, Ours vs Ours−G = 순수 검증 방식 효과.** 메모이제이션은 [4] 검색만 캐시해 검증 방식이 캐시로 새지 않음 (C6). top_k 공유는 '호출 1회당' 규칙임 (질문당 총 검색량은 §3 예산 지표로 병기).

## 2. 평가셋
| 셋 | 규모 | 원천 | 용도 |
|---|---|---|---|
| 단일홉 | 150 | KorQuAD dev ∩ 코퍼스 문서 (무변형) | 단순 질문 퇴행 확인 |
| 멀티홉 | 100~200 | 자체 구축 (아래 스키마) | 주력 |
| 오타 변형 | 30 | 위 질문에 자모 노이즈 주입 | 교정 계층 평가 |
| 데모 큐레이션 | 15 | 수동 (의도된 실패 3 포함) | 발표 시연 |

### 멀티홉 문항 스키마 (구축 시 필수 필드 — 이거 없으면 검증 평가 불가)
```json
{
  "qid": "mh_0001",
  "question": "베토벤이 태어난 도시가 속한 나라는?",
  "answer": "독일",
  "answer_aliases": ["독일", "Germany", "독일연방공화국"],
  "hops": [
    {"hop": 1, "relation": "bornIn", "intermediate_answer": "본",
     "intermediate_aliases": ["본", "Bonn"],
     "evidence_paragraph": "문단ID", "source_qa": "korquad_id_1"},
    {"hop": 2, "relation": "locatedIn", "intermediate_answer": "독일",
     "evidence_paragraph": "문단ID", "source_qa": "korquad_id_2"}
  ],
  "structure": "CHAIN"   // CHAIN | STAR | COMPARE
}
```
구축 절차: 브릿지 쌍 추출(W0.1) → 템플릿 결합 → LLM 패러프레이즈 → **shortcut 필터(Baseline이 Recall@10에 최종 정답 문단을 잡으면 탈락)** → 자연성 수검수.
금지: 자체 그래프 경로에서 역생성(순환 평가). 재배포 금지(CC BY-ND) — 재생성 스크립트만 공개.

## 3. 지표 정의
### 주지표 (검색 품질) — 유형별(단일홉/멀티홉/오타) 분해 산출
- Recall@5/10: 정답(또는 alias)이 포함된 gold 문단이 top-k에 존재하는 비율. 멀티홉은 hop별로도 산출
- MRR, nDCG@10
- **top-k 랭킹 리스트 구성 (C3 — 이 규칙 없이는 Ours/Ours−G/Agent-basic의 @k 계산 불가):**
  - 질문 수준 리스트 = 파이프라인이 실행한 **전체 경로A 벡터검색(재질의·분기 전부 포함) 결과 문단의 합집합을 벡터 유사도로 재랭킹**한 상위 k. 판정 gold = 최종 정답 문단.
  - 경로B(그래프) 문단은 이 리스트에서 **제외** → "graph-assist 구제 건수"로만 별도 집계 (벡터 점수 없어 MRR/nDCG 오염 방지).
  - hop별 Recall@k = 해당 hop의 **마지막 검색 호출** 결과 기준, gold = 해당 hop 중간정답 문단.
  - 검색은 metric 계산 위해 최소 max(k)=10 문단 반환 (06 top_k_paragraphs ≥ 10).
  - **예산 투명화:** 질문당 경로A 검색 호출 수·노출 문단 총수를 시스템 지표로 병기 (Ours 수십 vs Baseline 8).
### hop 수준 (차별화 증명)
- Intermediate EM: 중간 정답 일치율 (aliases 허용, 형태소 정규화)
- 오류 전파율: hop1 오답 → 최종 오답 전이 비율 (Ours vs Agent-basic 비교가 핵심 그림)
- 검증기 오답 탐지 P/R: 검증기가 기각한 후보 중 실제 오답 비율(P), 실제 오답 후보 중 기각된 비율(R). 검사별(type/name/desc/backlink) 기여 분해
### 의사결정 지표
- oracle@1 vs oracle@2 (candidates 로그 기반) → 조건부 분기 도입 판정 (ADR-8)
- graph-assist 구제 건수(경로B가 벡터 미스를 메꾼 수), backlink 차단 오답 건수, 재질의 성공 전환율, 분해 실패율, graph_miss율
### 시스템
- 질문당 LLM 콜 수·토큰 비용·latency p50/p95 (infra/llm.py 집계에서 자동 산출)

## 4. 목표
- **주 KPI (C2): 멀티홉 최종 Recall@10 — Agent-basic 대비 +X%p (X는 W4.3 1차 측정 후 확정, 통계검정 첨부) + Ours 절대치 병기.**
    Baseline 대비 수치는 참고용만 — §2 shortcut 필터가 Baseline을 판정자로 쓰므로 Baseline 최종 Recall≈0, "+25%p"는 구성상 자동 달성이라 헤드라인 금지.
- 분해 vs 검증 분리: Ours > Ours−G > Agent-basic 확인 (Ours−Ours−G = 순수 검증 기여, Ours−G−Agent-basic = 분해+이중검색 기여).
- 단일홉: Baseline 대비 퇴행 없음 (복잡한 파이프라인이 단순 질문을 해치지 않음을 증명)
- 오답을 고신뢰(confidence>0.8)로 내는 비율: Agent-basic 대비 절반 이하 (confidence 산식·Agent-basic 산출법은 03에서 동일 정의 선행 필요)
