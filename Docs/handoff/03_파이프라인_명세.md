## §0. 공용 데이터 객체

```python
PipelineState = {
  "request_id": str,
  "raw_question": str,                  # 불변
  "corrected_question": str,
  "corrections": [Correction],          # {"from","to","source":"L1|L2|L3|L4"}
  "goal_type_surface": str|None,        # 어미 판독 타입 (검산 전용)
  "hops": [Hop],
  "final_op": FinalOp|None,
  "hop_results": {int: HopResult},
  "mode": "FULL"|"SIMPLE",
  "answer": Answer|None,
  "trace": [TraceEvent],
}

Hop = {
  "id": int,
  "start": str,                         # 리터럴 or "{n}" 참조
  "start_node": str|None,               # 그라운딩 후 노드 ID
  "relation": str,                      # 스키마 10종 + "relatedTo"
  "relation_grounded": str|None,
  "direction": "fwd"|"inv",
  "expected": {
     "type": str,                       # PERSON|LOCATION|ORG|DATE|NUMBER|WORK|TERM
     "definition": str,                 # 조건의 재서술. 정답 고유명사 금지 (검증 8번)
  },
  "anchor_meta": {"name": str, "disambiguator": str|None},
  "sub_query": str,
  "hint_relations": [str]|None,         # 그라운딩 포기 시 실존 relation 목록
}

FinalOp = {"operator": "EARLIER|LATER|SAME|GREATER|SMALLER", "in": [int]}
# 연산자 요구 타입: EARLIER/LATER→DATE, GREATER/SMALLER→NUMBER, SAME→any
# yes/no 출력 연산의 답 포맷은 "yes"/"no" 고정

Candidate = {
  "entity": str,
  "origin": "vector"|"graph"|"both",
  "evidence_paragraph_ids": [str],
  "verification": Verification|None,
  "score": float,
}

Verification = {
  "type_check": bool|None,              # None = 약화모드 미시행
  "name_check": bool|None,
  "desc_check": bool,
  "backlink_check": "pass"|"absent"|"contradict"|None,   # 3값 (C1). None=약화모드 미시행
  "passed": bool,                       # 산식 아래 (C1)
  "weak_mode": bool,
}
# passed 규칙 (C1 — backlink 단독 실패가 pass_ratio에 묻히는 문제 해소):
#  1. backlink_check == "contradict" → passed=False (hard-veto, pass_ratio 무관)
#     "contradict" = 그래프에 동일 원본 relation의 '다른' tail이 실존 → 이 후보는 그 relation의 답이 아님
#  2. 그 외: backlink "absent"는 중립(시행수에서 제외), 나머지 시행 검사 통과율 ≥ pass_ratio
#  → 시나리오 B: '빈'은 모차르트-bornIn→잘츠부르크가 그래프에 실존하므로 "contradict" = veto (재질의로 회복)

HopResult = {
  "hop_id": int, "answer": str,
  "candidates": [Candidate],            # ★ 전량 보존 (oracle 분석 원천)
  "tie": bool, "branched": bool, "retries": int,
  "status": "OK"|"WEAK"|"FAILED",
  "queries_used": [str],
}

Answer = {
  "text": str,
  "status": "ANSWERED"|"PARTIAL"|"UNVERIFIABLE",
  "confidence": float,                  # 시행 검사 통과율의 hop 평균 + path_check 가산
  "path_check": bool|None,
  "evidence": [{"paragraph_id": str, "hop_id": int}],
  "verified_until": int|None,
}
```

## §1. 단계별 계약 (읽는 필드 → 쓰는 필드)

### [0] 교정 L1~L3 (결정적, LLM 금지)
- 읽기: raw_question / 의존: alias 사전, 약어 사전
- L1 규칙: NFC 정규화, 공백, 중복 조사, 약어 확장
- L2 사전 대조: 핵심 명사구가 전부 alias에 있으면 → 오타 없음 판정, L3·L4 스킵 플래그
- L3 오타 교정: 미매칭 토큰만. 후보 = alias 키 중 편집거리≤2 ∪ 자모거리 후보. **후보 유일 + 거리≤1일 때만 확정 교정.** 애매하면 후보 목록을 [1]에 전달
- 쓰기: corrected_question(잠정), corrections, 미해결 토큰+후보
- 실패 없음 (못 고치면 넘길 뿐)

### [1] 분해 통합 콜 (LLM 1콜, structured output)
- 프리필터: 관계 표지 0개 + 단문 → 콜 스킵, mode=SIMPLE
- 읽기: corrected_question(잠정), 미해결 토큰+후보, relation 스키마(프롬프트 상수)
- 쓰기: corrected_question(확정), corrections(L4), goal_type_surface, hops[], final_op
- 프롬프트 제약: relation은 스키마 내만 / 교정은 제시 후보 중 선택 or 원문 유지 + diff 강제 / start 참조 "{n}" / definition은 조건 재서술만·고유명사 금지 / fwd·inv 방향 few-shot 2개 필수
- 실패: 파싱 실패 → 재생성 1회 → SIMPLE

### [2] 계획 검증 8종 (코드, LLM 0콜)
| # | 검사 | 실패 처리 |
|---|---|---|
| 1 | relation ∈ 스키마+relatedTo | 재생성 1회 → SIMPLE |
| 2 | direction 유효 | 〃 |
| 3 | 참조 무결성(실존·DAG·op.in) — 위상정렬 1회 | 〃 |
| 4 | 깊이≤3 AND 총hop≤5 | 절단 or SIMPLE |
| 5 | 타입체인: leaf(또는 op출력) == goal_type_surface | 재생성 1회 |
| 6 | op 입력 타입 == 연산자 요구 타입 | final_op 제거 후 진행 |
| 7 | 병렬의 억지 직렬화 감지 | 참조 제거·분리 |
| 8 | definition 정답 누출: definition 내 **그 hop의 답(tail)·후속 hop 참조 엔티티**명 검출 (앵커·조건 서술의 기존 엔티티는 허용 — 예 "베토벤이 태어난 도시"의 '베토벤'은 OK, C5) | 재생성 1회 |
- 쓰기: execution_order(위상정렬 결과)

### [3] 그라운딩 (LLM 0~1콜)
- anchor: alias exact → (복수 시 disambiguator·노드설명 유사도) → 임베딩 top-1. 미해결 → graph_miss 마킹, 진행
- relation: exact(대부분) → 스냅(cos>θ_high) → 타이브레이크(θ_low~θ_high, 객관식 1콜) → 포기(<θ_low): relation_grounded=None + hint_relations 첨부
- 쓰기: start_node, relation_grounded, hint_relations

### [실행기] ready-set 루프
```python
async def execute(state):
    while 미완료 hop 존재:
        ready = [h for h in hops if h.id not in done and start_해석가능(h, done)]
        results = await gather(*[resolve_hop(h, state) for h in ready])
        done.update(results)
    # tie 발생 시: 후속 hop을 chain_id별 복제, 활성 체인 ≤3
```
- "{n}" 치환은 실행 시점. 재귀 함수 구현 금지 (ADR-3)

**[4] 검색 (LLM 1콜):**
- 경로A: Chroma(sub_query [+ definition 결합 임베딩 — HyDE]) top-8 문단 → LLM 1콜 후보 추출
- 경로B: graph.neighbors(start_node, relation_grounded, direction) → 노드 + 근거 문단ID
- 병합: alias 기준 동일 엔티티 → origin="both". graph 출신은 backlink="pass" 기록(자동통과)하되 **"contradict" 검사는 면제하지 않음** — 스냅 오염 후보가 veto를 우회하는 구멍 차단 (C1/ADR-6 2겹 그물 유지)
- 후보 0 → 즉시 [6]

**[5] 검증·확정 (LLM 0콜):**
- type: node.type == expected.type (direction 반영)
- name: 후보 ∈ alias
- backlink: inv = inverse(**분해 의도 원본 relation**) (grounded 아님, ADR-6). 3값 판정 (C1):
    · has_edge(후보→start_node, inv) → "pass"
    · 엣지 없음 && ∃ t'≠후보 로 has_edge(t'→start_node, inv) → "contradict" (hard-veto)
    · 엣지 없음 && 그런 t' 없음 → "absent" (중립, 시행수 제외)
- desc: cos(후보 근거문단 or 노드 설명, expected.definition) > θ_desc
- graph_miss → weak_mode(type/name/backlink 생략 가능한 것만 시행)
- score = 통과수 + λ·벡터유사도. top1 passed → 확정. margin<ε → tie. shortcut: 문단에서 최종 leaf 타입 답 동시 검출 시 잔여 hop 스킵 플래그
- 전원 미달 → [6]

**[6] 재질의 (LLM 1콜, hop당 max 2):**
- 프롬프트 주입: 실패 사유(검사별) + queries_used(중복 금지) + hint_relations/이웃 relation 목록
- 소진 → status=FAILED, 체인 중단

### [7] final_op (코드)
- 날짜/숫자 정규화 후 연산. 파싱 실패 → PARTIAL로 강등(두 값 나열)

### [8] Path-Check + 체인 중재 (코드)
- path_exists(anchor₀, [의도 relation들], answer) — 제약 BFS (RoG Algorithm 1 방식)
- 존재 → confidence 가산 + 뱃지 / 부재 → 중립 (감점 없음)
- 체인 복수 → 검증 점수 합 + path_check로 승자 결정

### [9] 응답 조립 (LLM 1콜)
- ANSWERED / PARTIAL / UNVERIFIABLE("hopN까지 확인: {중간결과}, 부족: {사유}", verified_until)

### [로깅]
- 전 단계 진입/이탈 TraceEvent. 필수: correction_applied, decompose_ok/fallback, plan_validation_fail(#), grounding_snap/tiebreak/miss, candidates_merged, verify_result, tie_detected, requery(사유), shortcut, path_check, answer_status
- candidates 전량 보존 (ADR-8)

## §2. 모드 매트릭스
| mode | 경로 | 용도 |
|---|---|---|
| FULL (Ours) | 전 단계 (verifier=quad) | 본 시스템 |
| SIMPLE | [0]→경로A 1회→[9] | 분해 불가/불필요 질문 폴백 |
| Agent-basic | [0]→**원질의(분해 없음)** 경로A→llm_judge→재검색→[9] | 현실 대조군 "타 팀 공통선" (분해 미사용 → verifier 주입 구조 아님, C4) |
| Ours−G (ablation) | FULL과 동일, **verifier=llm_judge**만 교체 (verifier 주입) | 이득이 분해가 아닌 검증 방식임을 분리 증명. 분해 사용 (ADR-4, C4) |
| Baseline | 원질의→경로A 1회→[9] | 기준선 |
