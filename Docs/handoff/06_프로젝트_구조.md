# 06. 프로젝트 구조 & 설정

구현은 병렬 멀티에이전트로 진행한다(ADR-10). 이 문서의 핵심은 디렉토리 그림이 아니라 **모듈 계약표** — 에이전트 하나가 딴 파일을 안 열고도 자기 모듈을 짤 수 있게 만드는 계약이다.

## 디렉토리

```
configs/
  settings.yaml            # 모델·임계값·상한. 수정 = 실험 = 커밋
  relations.yaml           # relation 스키마 (W0.2에서 확정)
  prompts/                 # 프롬프트를 코드에서 분리 (튜닝 = 설정 diff)
    decompose.txt  extract_candidates.txt  requery.txt  tiebreak.txt  triple_extract.txt
src/verihop/
  settings.py              # yaml → pydantic Settings. yaml 로드는 여기 한 곳뿐
  models.py                # §0 데이터 객체 (전 모듈 공용 계약)
  ports.py                 # Protocol 5개 — usecase가 보는 유일한 외부 계약
  domain/                  # 순수 판단 로직. stdlib + models만 import (I/O 없음)
    plan_rules.py          # 계획 검증 8종
    verify_rules.py        # Quad 검증 판정 + passed 산식 (C1)
  usecases/                # 오케스트레이션. ports를 통해서만 외부 접근
    correct.py             # [0]
    decompose.py           # [1]
    validate_plan.py       # [2] (domain.plan_rules 호출)
    ground.py              # [3]
    resolve_hop.py         # [4][5][6] 캡슐화 (search/verify/reformulate)
    finalize.py            # [7][8][9]
    executor.py            # ready-set 루프 + 분기
  adapters/                # ports 구현. 서드파티(openai/chroma/networkx)는 여기서만
    openai_llm.py          # LLMPort. 재시도·캐싱·비용집계 단일 관문
    openai_embedder.py     # EmbedderPort
    faiss_index.py         # VectorIndexPort (ADR-11: Chroma→FAISS)
    networkx_graph.py      # GraphPort
    json_tracer.py         # TracerPort
    memo_cache.py          # [4] 검색결과 캐시 데코레이터 (C6)
  bootstrap.py             # 조립 루트. build_pipeline(mode)가 유일한 조립 지점
apps/
  cli.py                   # 단일 질문 디버깅 진입점
  streamlit_app.py         # 데모 UI
scripts/                   # w0_yield_check / build_corpus / build_index / build_graph / build_multihop_set
eval/                      # run_eval / metrics / report
tests/
  fakes.py                 # 포트별 가짜 구현 (usecase 테스트 주입용)
  unit/                    # domain 순수 함수 테스트 (mock 없음)
  usecase/                 # usecase 테스트 (fakes 주입)
tools/
  check_layers.sh          # 계층 의존 규칙 grep 게이트
data/                      # gitignore (코퍼스·pickle·로그·results)
```

## 계층 의존 규칙

```
domain  ←  usecases  ←  adapters  ←  bootstrap / apps / scripts
```

- 화살표 방향으로만 의존한다.
- usecases는 adapters를 import하지 않는다. 검사: `grep -r "verihop.adapters" src/verihop/usecases/` 가 빈 출력.
- usecases는 구체 구현(openai/chromadb/networkx)을 직접 부르지 않는다. ports만 본다.
- domain은 stdlib과 `verihop.models` 외에는 아무것도 import하지 않는다.
- 이 규칙을 `tools/check_layers.sh`가 자동 검사한다. 통과가 모든 모듈의 done 기준.

## ports (seam 5개, 상한 5개)

usecase가 보는 외부 세계는 이 5개가 전부다. 6번째가 필요하면 02에 ADR을 먼저 쓴다(ADR-10).

| Port | 용도 | 구현 |
|---|---|---|
| LLMPort | 구조화 출력 LLM 호출. 재시도·캐싱·비용집계 내부화 (전 콜이 통과) | openai_llm |
| EmbedderPort | 임베딩 + 캐시 | openai_embedder |
| VectorIndexPort | 경로A 벡터 검색 (top_k ≥ 10) | faiss_index (ADR-11: Chroma→FAISS) |
| GraphPort | 경로B 이웃·엣지·타입·relation 조회 | networkx_graph |
| TracerPort | 구조화 로그 (request_id 관통) | json_tracer |

verifier(quad ↔ llm_judge)는 포트가 아니라 resolve_hop에 주입하는 함수다(ADR-4). Ours−G arm이 이 주입으로 생긴다.

## 모듈 계약표 (에이전트 작업 단위)

각 행이 에이전트 하나의 작업 계약이다. reads/writes는 models.py 타입, may-import는 허용 목록, test는 done을 정의하는 파일.

| 모듈 | reads | writes | may-import | test |
|---|---|---|---|---|
| domain/plan_rules | Hop[], FinalOp, goal_type_surface | 검증 결과(8종 pass/fail + 처리) | models | unit/test_plan_rules |
| domain/verify_rules | 검사 4종 결과, thresholds | Verification(passed 산식·C1 veto) | models | unit/test_verify_rules |
| usecases/correct | raw_question, alias·약어 사전 | corrected(잠정), corrections, 미해결 토큰 | models, ports(Embedder) | usecase/test_correct |
| usecases/decompose | corrected(잠정), 후보, relation 스키마 | hops[], final_op, goal_type_surface, corrected(확정) | models, ports(LLM) | usecase/test_decompose |
| usecases/validate_plan | hops[], final_op | execution_order 또는 SIMPLE 강등 | models, domain.plan_rules | usecase/test_validate_plan |
| usecases/ground | hops[], alias, graph relation 집합 | start_node, relation_grounded, hint_relations | models, ports(Graph, Embedder, LLM) | usecase/test_ground |
| usecases/resolve_hop | Hop, state, verifier | HopResult(candidates 전량) | models, ports(LLM, Vector, Graph), domain.verify_rules | usecase/test_resolve_hop |
| usecases/finalize | hop_results, final_op | Answer, path_check | models, ports(LLM, Graph) | usecase/test_finalize |
| usecases/executor | hops[], execution_order | hop_results, 체인 분기 | models, usecases.resolve_hop | usecase/test_executor |
| adapters/* | (각 Port 계약) | (각 Port 계약) | 해당 서드파티 | usecase 테스트에서 fakes로 대체 |
| bootstrap | Settings, mode | 조립된 파이프라인 | 전 계층 | — |

캐시 주의(C6): resolve_hop의 memoize는 [4] 검색결과만 캐시한다. 검증·재질의는 매번 재실행(quad=0콜이라 비용 손실 없음). verifier·hop_id·retries·FAILED를 캐시에 박으면 3단 비교/재평가가 오염된다.

## 설정 단일화 5규칙

1. yaml 로드 지점은 settings.py 하나. 다른 모듈이 직접 yaml을 열지 않는다.
2. 전역 import 대신 생성자/인자 주입. 값은 위에서 내려온다.
3. 프롬프트도 설정이다(configs/prompts/). 튜닝이 코드 diff가 아니라 설정 diff가 되도록.
4. 실험 = settings.yaml 수정 + 커밋. git이 실험 기록을 담당한다.
5. 환경변수는 OPENAI_API_KEY 하나만.

## 과설계 방지선 (에이전트 작업 규칙)

방치하면 에이전트가 습관적으로 들여오는 것들. 하지 않는다:

- DI 프레임워크 없음. bootstrap.py의 평범한 함수로 조립한다.
- usecase는 함수다. 클래스가 아니다.
- 레포지토리·이벤트버스·CQRS 없음.
- Protocol은 5개가 상한. 늘리려면 ADR(ADR-10).
- 추상화는 구현이 2개 이상일 때만. 구현 1개짜리 인터페이스를 만들지 않는다(ports 5개는 런타임 다형성이 아니라 에이전트 계약 격리가 목적 — ADR-10 참조).

## configs/settings.yaml 초안

```yaml
llm:
  model: gpt-4o-mini
  temperature: 0.0
  max_retries: 2
embedding:
  model: text-embedding-3-small     # W1.2 스팟체크 미달 시 한국어 특화로 교체
retrieval:
  top_k_paragraphs: 10               # Recall@10 계산 위해 ≥ max(k). LLM 후보추출도 이 상한 (C3)
  use_hyde_definition: true          # sub_query + expected.definition 결합 임베딩
grounding:
  theta_high: 0.90                   # 스냅 확정 (무보정값 — W1.2에서 relation 임베딩 실측 보정 필요)
  theta_low: 0.75                    # 미만이면 포기 + 힌트
verification:
  pass_ratio: 0.75                   # 시행 검사 통과 비율. 단 backlink="contradict"는 pass_ratio 무관 hard-veto (03 §0/C1)
  theta_desc: 0.60
  score_lambda: 0.3                  # 벡터 유사도 가중
execution:
  max_requery: 2
  max_depth: 3
  max_hops_total: 5
  tie_epsilon: 0.15                  # margin < ε → tie
  max_active_chains: 3
  memoize: true                      # [4] 검색결과만 캐시 (C6)
paths:
  corpus: data/corpus.jsonl
  chroma_dir: data/chroma
  graph_pickle: data/graph.pkl
  alias: data/alias.json
```

## configs/relations.yaml 초안 (W0.2 분포 실측 후 확정 — 아래는 가설)

```yaml
# head_type --relation--> tail_type. direction=fwd가 이 방향.
relations:
  - {name: bornIn,     head: PERSON,   tail: LOCATION, ko: ["태어난", "출생지", "출신"]}
  - {name: diedIn,     head: PERSON,   tail: LOCATION, ko: ["사망한", "죽은 곳"]}
  - {name: locatedIn,  head: LOCATION, tail: LOCATION, ko: ["속한", "위치한", "있는"]}
  - {name: capitalOf,  head: LOCATION, tail: LOCATION, ko: ["수도"]}
  - {name: nationality,head: PERSON,   tail: LOCATION, ko: ["국적", "국가의"]}
  - {name: studiedAt,  head: PERSON,   tail: ORG,      ko: ["공부한", "졸업한", "출신 학교"]}
  - {name: memberOf,   head: PERSON,   tail: ORG,      ko: ["소속", "가입한"]}
  - {name: createdBy,  head: WORK,     tail: PERSON,   ko: ["만든", "작곡한", "쓴", "감독한"]}
  - {name: teacherOf,  head: PERSON,   tail: PERSON,   ko: ["스승", "제자", "가르친"]}
  - {name: occupation, head: PERSON,   tail: TERM,     ko: ["직업", "활동한 분야"]}
fallback: relatedTo        # 스키마 밖 → 검증 약화 모드(backlink 생략)
types: [PERSON, LOCATION, ORG, DATE, NUMBER, WORK, TERM]
operators:
  EARLIER: {in_type: DATE}
  LATER:   {in_type: DATE}
  GREATER: {in_type: NUMBER}
  SMALLER: {in_type: NUMBER}
  SAME:    {in_type: any, out: yesno}
```
