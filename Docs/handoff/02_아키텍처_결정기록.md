# 02. 아키텍처 & 의사결정 기록 (ADR)

구현 중 "왜 이렇게 되어 있지?"가 나오면 여기를 먼저 본다. 결정을 뒤집으려면 해당 ADR의 근거를 반박해야 한다.

## 아키텍처 전경
```
Streamlit UI
   │ (진행 상황 스트리밍)
LangGraph 상태 그래프
   [교정 L1~L3] → [분해+분석 LLM 1콜] → [계획 검증 8종] → [그라운딩]
   → 실행기(ready-set 루프) { resolve_hop = 검색(A벡터+B그래프) → Quad 검증 → 실패 시 힌트 재질의 }
   → [final_op(결정적 비교)] → [Path-Check] → [응답 조립]
   │
   ├─ ChromaDB: 문단 벡터 인덱스
   ├─ NetworkX: 지식그래프 (GraphPort 뒤) + alias 사전
   └─ JSON 구조화 로그 (request_id 관통)
```

## ADR-1. 검토를 LLM judge가 아닌 그래프 조회로
- 문제: 오답을 생성하는 모델과 심사하는 모델이 같으면 그럴듯한 오답이 그대로 통과하고, 탈락 사유도 남지 않는다.
- 결정: type/name/backlink는 그래프와 사전 조회로 결정적으로 판정하고(LLM 0콜), desc만 임베딩 유사도로 본다.
- 근거: EMNLP'25 서베이 분류에서 KG를 "Validator"로 쓰는 자리이고, 아직 덜 개척된 포지션이다. 발표에서는 이 접근을 차별점으로 내세운다.

## ADR-2. 질문 표현 = hop 원자 + start 참조 (유형 분기 없음)
- 여정: 처음에는 3유형으로 분류했다가, 인지과학 5축(Lehnert/Graesser)을 거쳐 데이터셋 기준으로 감량하고, CHAIN/STAR/PARALLEL 위상 3형으로 좁혔다. 마지막에 모든 형태에서 relation·type 비교는 같고 시작점만 다르다는 통찰로 하나로 통일했다.
- 결정: hop = (start, relation, direction, expected{type,definition}, sub_query). start가 리터럴이면 앵커이고, `{n}`이면 hop n의 답을 참조한다. 직렬·팬아웃·병렬·혼합이 모두 참조만으로 표현된다. 답은 아무도 참조하지 않는 leaf이며, final_op가 있으면 그 출력이 답이 된다.
- 이론 근거: QDMR(Break, TACL 2020)의 SELECT/PROJECT/#참조와 동형이다. 13개 연산자 중 결정적 비교만 final_op로 지원하고, FILTER/AGGREGATE류는 범위 밖으로 둔다.

## ADR-3. 실행 = 개념은 재귀, 구현은 ready-set 반복
- 결정: resolve(start, relation)→t 를 공통 원자로 삼아, 의존성이 해소된 hop부터 반복 실행한다(asyncio.gather).
- 재귀 함수로 구현하지 않는 이유는, 팬아웃이나 분기에서 병렬성을 잃고 재시도와 부분 실패가 콜스택에 얽혀 trace가 불가능해지기 때문이다.
- 부수 효과로 병렬성이 공짜로 따라오고, 유형별 실행기 코드가 사라진다.

## ADR-4. resolve 단일화가 여는 재사용 최적화 (설계에 내장)
- 메모이제이션: [4] 경로A 검색 결과만 캐시한다(키는 (start_resolved, relation_grounded, direction, sub_query, definition)). 검증과 재질의는 매번 다시 실행하는데, quad가 0콜이라 비용 손실이 없다. 검증 결과, verifier, hop_id, retries, FAILED를 캐시에 박으면 3단 비교와 재평가가 증상 없이 오염되므로 금지한다(C6). **키에서 direction을 빠뜨리면 안 된다.**
- verifier 주입: resolve(verifier=quad|llm_judge) 형태로 Ours−G ablation arm(FULL 파이프라인 + llm_judge)을 공짜로 구현할 수 있고, 이렇게 하면 이득이 분해가 아니라 검증 방식에서 온다는 것을 분리해서 증명할 수 있다. Agent-basic은 분해가 없어 이 구조로는 만들 수 없으므로 별도의 현실 대조군으로 둔다(05 §1/C4).
- 앵커 호이스팅, 배칭, 프롬프트 캐싱은 Day 11+ 튜닝 항목으로 미룬다.
- 메모이제이션과 verifier 주입은 나중에 붙이기 어려우므로 처음부터 넣는다.

## ADR-5. 그래프 저장 = NetworkX (Neo4j 아님)
- 근거: 연산이 전부 1-hop 이웃, 엣지 존재, 속성, 제약 BFS라 인메모리에서 O(1)급이다. 노드는 수만 규모이고 단일 프로세스로 돈다. 램 초과나 패턴 쿼리, 동시 접근처럼 Neo4j가 유리해지는 조건에 해당하지 않는다. 2주 일정에 Neo4j는 인프라 리스크만 더한다.
- 안전장치로 GraphPort(Protocol)로 감싸 두어 교체할 수 있게 한다. Chroma는 문단 벡터만 전담하고 이웃 연산은 하지 않는다. 역할을 혼동하지 않는다.

## ADR-6. relation 그라운딩 = 무학습 런타임 정렬 (RoG 파인튜닝 대체)
- 문제: RoG는 KG 어휘를 파인튜닝으로 증류하므로 KG마다 다시 학습해야 한다. ToG는 홉마다 prune LLM 콜을 써서 비싸다.
- 결정: 분해가 relation을 먼저 내놓으므로(계획이 선행한다) anchor 노드의 실존 엣지 집합에 정렬만 한다. exact 매칭을 먼저 보고, 안 되면 임베딩으로 스냅한다(>θ_high). θ_low~θ_high 구간이면 LLM 타이브레이크로 객관식 1콜을 쓰고, <θ_low이면 포기하면서 실존 relation 목록을 재질의 힌트로 넘긴다.
- 함정과 방어: bornIn↔diedIn은 임베딩상 가까워서, 이중 임계값으로 타이브레이크를 유도한다. 여기에 더해 **backlink 검증을 grounded가 아니라 분해가 의도한 원본 relation 기준으로** 하기 때문에, 스냅이 오염돼도 회수된다. 그물이 2겹인 셈이다.

## ADR-7. 교정 = 4계층, LLM 자유 교정 금지
- L1은 규칙 교정으로 NFC, 공백, 약어를 처리한다. L2는 alias 대조로, 정상 질문이면 여기서 조기 종료한다. L3은 편집거리와 자모거리를 쓰되 **후보를 그래프 실존 엔티티로 제한**해서 과교정을 구조적으로 막는다. L4는 분해 콜에 통합한 객관식으로, 후보 중 하나를 고르거나 원문을 유지하며 diff를 강제한다.
- alias 사전(위키 redirect)이 교정, 링킹, name 검증의 3역을 겸한다.

## ADR-8. hop 다답 = top-1 기본 + margin 조건부 분기, 도입은 측정 후
- 문제: top-1 greedy는 과거 hop을 잘못 골랐을 때 복구할 경로가 없다. 재질의는 현재 hop만 고치기 때문이다. SOTA도 복수 경로를 유지한다(ToG는 빔, RoG는 top-3).
- 결정: 기본은 top-1이고, margin<ε이거나 신뢰도가 낮으면 top-2로 분기하며, 활성 체인은 3개까지 유지한다(≤3). 체인 간 승자는 검증 점수의 합으로 정하고 Path-Check가 중재한다.
- 도입 판정: Day 8에 candidates를 전량 로그로 남겨 oracle@1과 @2를 실측한다. 갭이 크면 구현하고(반나절), 작으면 top-1을 유지한다. **candidates를 top-1만 남기고 버리면 이 분석 자체가 불가능해지므로 전량 보존한다.**
- 부수 효과로 동명이인 해소가 이 메커니즘으로 함께 풀린다.

## ADR-9. 분해 메타데이터 확장 + 정답 누출 방지
- 결정: hop에 expected.definition(예: "베토벤이 태어난 도시", 조건을 다시 서술한 것)과 anchor_meta{name, disambiguator}를 추가한다.
- 용도: desc 검증을 후보 설명문과 기대 정의를 비교하는 방식으로 정밀화하고, sub_query와 definition을 결합한 임베딩으로 검색을 보강하며(HyDE 패턴), disambiguator로 앵커를 링킹한다.
- 금지: definition에는 답이 될 고유명사를 넣지 않는다. 이를 어기면 LLM의 사전지식이 검증을 오염시킨다. 답이 맞으면 검증이 무의미해지고, 틀리면 오답을 오히려 강화한다. 프롬프트 규칙과 코드 검사를 함께 둔다(계획 검증 8번에서 definition 안에 alias 엔티티명이 검출되면 재생성한다).

## ADR-10. 4계층 + ports 5개 (core/infra 2분할 → 클린 아키텍처)
- 배경: 이 프로젝트는 1인 개발이지만 구현은 병렬 멀티에이전트로 진행한다. 에이전트는 전체 설계를 머릿속에 들고 있지 않아서, 편하면 계층 경계를 넘고(usecase에서 adapter를 직접 호출), 방치하면 레포지토리나 팩토리, CQRS를 습관적으로 들여온다. 사람 한 명이면 과설계일 명시적 경계가, 병렬 에이전트에게는 값싼 보험이 된다.
- 결정: 계층은 domain ← usecases ← adapters ← bootstrap/apps/scripts 4계층으로 두고, 의존은 화살표 방향으로만 흐른다. seam은 ports.py의 Protocol 5개(LLMPort/EmbedderPort/VectorIndexPort/GraphPort/TracerPort)로 고정하고, usecase는 이 5개만 본다. 구체 구현이 있다는 것조차 모른다.
- 이전 결정 뒤집기: ADR-5의 "Protocol은 GraphStore 하나만(과설계 금지)"을 5개로 확대한다. 근거를 반박하는 것이 아니라 전제가 바뀐 것이다. GraphStore 1개 결정은 런타임 다형성을 기준으로 삼았고(구현이 하나면 Protocol이 필요 없다), ADR-10은 에이전트가 구현을 읽지 않고 계약만 읽는 격리를 기준으로 삼는다. 파이썬 테스트는 덕타이핑으로 Protocol 없이도 되지만, ports.py 한 파일이 모든 seam을 열거하면 그것이 에이전트의 컨텍스트 지도가 된다.
- 상한과 방어: **포트 5개가 천장이다. 6번째를 추가하려면 이 문서에 ADR을 먼저 쓴다.** 에이전트는 반드시 포트를 늘리려 하므로 이 천장이 규율의 핵심이다. 계층 위반은 tools/check_layers.sh(grep 게이트)가 자동으로 검출하고, 커밋 전에 통과하는 것이 done 기준이다.
- 함정: 계약을 산문으로만 적으면 에이전트 A와 B가 같은 모듈을 다르게 구현한다. 에이전트는 문장은 무시해도 실패하는 테스트는 따른다. 그래서 06의 모듈 계약표는 시그니처(models.py 타입)와 import 허용목록, 테스트 파일까지 실행 가능한 형태로 못박는다.

## 폐기된 대안 (재논의 방지)
- Neo4j 도입
- DAG 전용 스키마와 유형별 실행기
- KorQuAD 브릿지 데이터 재배포
- 그래프 경로에서 평가 문항을 역생성하는 방식(순환 평가)
- 5축 전체 태깅
- 전면 빔서치
- 2Wiki를 메인으로 삼는 안. 비교 수치상 이점은 있었지만 한국어와 자체 구축을 위해 되돌렸고, 대신 hop GT는 자체 평가셋이 책임진다.
