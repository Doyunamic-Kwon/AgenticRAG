# 05. 평가 명세

## 1. 3단 비교 (동일 질문셋, 동일 코퍼스)
| 모드 | 구성 | 존재 이유 |
|---|---|---|
| Baseline | 원질의 1회 벡터 검색 (분해·검증 없음) | 문제가 존재함을 보여주는 용도. KPI 대조군은 아니다 (§2 shortcut 필터의 판정자라 최종 Recall≈0, C2) |
| Agent-basic | 교정 + 원질의(분해 없음) LLM judge 검토 + 재검색 | 현실 대조군(타 팀 공통선). **주 KPI 비교선** |
| Ours−G (ablation) | Ours 전체 파이프라인 + verifier=llm_judge (verifier 주입) | 이득이 분해가 아니라 그래프 검증에서 온다는 점을 분리해 증명 (거의 무료, ADR-4/C4) |
| Ours | + 분해 + 그라운딩 + Quad/Path 검증 + 힌트 재질의 | 제안 방식의 효과를 확인하는 대상 |

공정성 원칙은 네 모드가 검색 코드와 인덱스, top_k를 공유하는 것이다. Ours와 Agent-basic의 비교는 파이프라인 전체 효과를, Ours와 Ours−G의 비교는 순수 검증 방식의 효과를 본다. 메모이제이션은 [4] 검색만 캐시하므로 검증 방식이 캐시로 새지 않는다 (C6). top_k 공유는 호출 1회당 규칙이며, 질문당 총 검색량은 §3 예산 지표로 병기한다.

## 2. 평가셋
| 셋 | 규모 | 원천 | 용도 |
|---|---|---|---|
| 단일홉 | 150 | KorQuAD dev ∩ 코퍼스 문서 (무변형) | 단순 질문 퇴행 확인 |
| 멀티홉 | 100~200 | 자체 구축 (아래 스키마) | 주력 |
| 오타 변형 | 30 | 위 질문에 자모 노이즈 주입 | 교정 계층 평가 |
| 데모 큐레이션 | 15 | 수동 (의도된 실패 3 포함) | 발표 시연 |

### 원칙: 평가셋에는 gold만, 예측은 런 출력에

예측값을 평가셋 파일에 넣지 않는다. 예측은 모드(baseline/basic/ours_g/ours)와 실행마다 달라지므로, 평가셋에 넣는 순간 특정 런에 종속된 파일이 된다. 평가셋은 불변 gold로 고정되어야 4모드 비교가 공정하다. 따라서 구조를 이렇게 나눈다.

- `data/eval/multihop.jsonl` — gold (불변)
- `results/{run_id}/per_question.jsonl` — 예측 (런마다 생성)
- 채점기는 qid와 hop 인덱스로 조인해 비교한다.

### gold 스키마 (멀티홉 문항, 구축 시 필수 필드)

기존 스키마에는 relation만 있고 start(앵커), direction, 노드 type이 없었다. 분해 정확도와 type check 채점에 모두 필요한 값이라 보강한다.

```json
{
  "qid": "mh_0001",
  "question": "베토벤이 태어난 도시가 속한 나라는?",
  "answer": "독일",
  "answer_aliases": ["독일", "Germany"],
  "structure": "CHAIN",                    // CHAIN | STAR | COMPARE
  "hops": [
    {"hop": 1,
     "start": "베토벤",                    // gold 앵커
     "start_type": "PERSON",
     "relation": "bornIn",
     "direction": "fwd",
     "target": "본",                       // 노드 name (기존 intermediate_answer)
     "target_type": "LOCATION",            // type check 채점 기준
     "target_aliases": ["본", "Bonn"],
     "evidence_paragraph": "문단ID",
     "source_qa": "korquad_id_1"},
    {"hop": 2,
     "start": "{1}",                       // 참조 표기 — 분해 구조 채점용
     "start_type": "LOCATION",
     "relation": "locatedIn", "direction": "fwd",
     "target": "독일", "target_type": "LOCATION",
     "target_aliases": ["독일", "Germany"],
     "evidence_paragraph": "문단ID", "source_qa": "korquad_id_2"}
  ]
}
```

definition은 gold에 넣지 않는다. definition은 시스템이 생성하는 기대 프로필(수단)이지, 정답이 유일하게 존재하는 대상이 아니다. "베토벤이 태어난 도시"와 "베토벤의 출생 도시"가 둘 다 맞으므로 EM 채점이 불가능하다. definition의 품질은 gold 대조가 아니라 정답 누출 검사(계획 검증 8번)로만 관리한다.

### 런 출력 스키마 (예측 기록, 신규)

```json
// results/{run_id}/per_question.jsonl (1행 = 1질문)
{
  "qid": "mh_0001",
  "mode": "ours",
  "predicted_plan": {                      // 분해 산출물 그대로 덤프
    "hops": [{"id": 1, "start": "베토벤", "relation": "bornIn", "direction": "fwd",
              "expected_type": "LOCATION", "definition": "...", "sub_query": "..."}],
    "final_op": null
  },
  "hop_results": [{                        // 실행 산출물
    "hop_id": 1, "resolved_answer": "본",
    "relation_grounded": "bornIn",
    "candidates": [ ... 전량 ... ],         // oracle 분석 원천
    "verification_per_candidate": [ ... ],
    "retries": 1, "status": "OK"
  }],
  "final_answer": "독일", "confidence": 0.88, "path_check": true,
  "retrieved_paragraphs_per_hop": {"1": ["p_12", "p_44"], "2": ["p_63"]}
}
```

### 채점 조인 규칙

| 채점 항목 | gold 필드 | 예측 필드 | 방식 |
|---|---|---|---|
| 분해: relation 정확도 | hops[i].relation, direction | predicted_plan.hops[i] | hop 인덱스 정렬 후 EM |
| 분해: 구조 정확도 | start 참조 패턴 | predicted_plan 참조 패턴 | 참조 그래프 동형 비교 |
| type check 채점 기준 | target_type | expected_type와 검증 결과 | 검증기 P/R 산출의 정답지 |
| 중간 정답 | target(+aliases) | resolved_answer | 정규화 후 EM |
| 검색 | evidence_paragraph | retrieved_paragraphs_per_hop | Recall@k |
| oracle@k | target | candidates 랭킹 | 포함 순위 |

hop 수가 gold와 예측에서 다르면(과분해 또는 저분해) 정렬 불가로 처리하고 분해 실패로 집계한다. 이것 자체가 분해 정확도 지표의 일부다.

구축 절차는 브릿지 쌍 추출(W0.1), 템플릿 결합, LLM 패러프레이즈, shortcut 필터, 자연성 수검수 순으로 진행한다. shortcut 필터는 Baseline이 Recall@10에서 최종 정답 문단을 잡으면 해당 문항을 탈락시킨다.

자체 그래프 경로에서 역생성하는 것은 순환 평가가 되므로 금지한다. 재배포도 금지이며(CC BY-ND), 재생성 스크립트만 공개한다.

## 3. 지표 정의
### 주지표 (검색 품질)
단일홉·멀티홉·오타 유형별로 분해해서 산출한다.

- Recall@5/10: 정답(또는 alias)이 포함된 gold 문단이 top-k에 존재하는 비율. 멀티홉은 hop별로도 산출한다.
- MRR, nDCG@10
- top-k 랭킹 리스트 구성 (C3, 이 규칙이 없으면 Ours/Ours−G/Agent-basic의 @k를 계산할 수 없다):
  - 질문 수준 리스트는 파이프라인이 실행한 전체 경로A 벡터검색(재질의·분기 전부 포함) 결과 문단의 합집합을 벡터 유사도로 재랭킹한 상위 k다. 판정 gold는 최종 정답 문단.
  - 경로B(그래프) 문단은 이 리스트에서 제외하고, "graph-assist 구제 건수"로만 별도 집계한다. 벡터 점수가 없어 MRR/nDCG 오염을 막기 위함이다.
  - hop별 Recall@k는 해당 hop의 마지막 검색 호출 결과를 기준으로 하며, gold는 해당 hop 중간정답 문단이다.
  - 검색은 metric 계산을 위해 최소 max(k)=10 문단을 반환한다 (06 top_k_paragraphs ≥ 10).
  - 예산 투명화를 위해 질문당 경로A 검색 호출 수와 노출 문단 총수를 시스템 지표로 병기한다 (Ours 수십 회 대 Baseline 8회).

### hop 수준 지표
- Intermediate EM: 중간 정답 일치율 (aliases 허용, 형태소 정규화)
- 오류 전파율: hop1 오답이 최종 오답으로 전이되는 비율 (Ours와 Agent-basic 비교가 핵심 그림)
- 검증기 오답 탐지 P/R: 검증기가 기각한 후보 중 실제 오답 비율(P), 실제 오답 후보 중 기각된 비율(R). 검사별(type/name/desc/backlink) 기여로 분해한다.

### 의사결정 지표
- oracle@1과 oracle@2 비교 (candidates 로그 기반)로 조건부 분기 도입을 판정한다 (ADR-8).
- graph-assist 구제 건수(경로B가 벡터 미스를 메꾼 수), backlink 차단 오답 건수, 재질의 성공 전환율, 분해 실패율, graph_miss율

### 시스템
- 질문당 LLM 콜 수·토큰 비용·latency p50/p95 (adapters/openai_llm.py 집계에서 자동 산출)

## 4. 목표
- **주 KPI (C2)는 멀티홉 최종 Recall@10이다.** Agent-basic 대비 +X%p로 잡되(X는 W4.3 1차 측정 후 확정하고 통계검정을 첨부한다), Ours 절대치를 함께 병기한다. Baseline 대비 수치는 참고용으로만 쓴다. §2 shortcut 필터가 Baseline을 판정자로 쓰기 때문에 Baseline 최종 Recall은 0에 가깝고, "+25%p"는 구성상 자동으로 달성되므로 헤드라인으로 쓰지 않는다.
- 분해와 검증의 분리: Ours > Ours−G > Agent-basic 순서를 확인한다. Ours에서 Ours−G를 뺀 값이 순수 검증 기여이고, Ours−G에서 Agent-basic을 뺀 값이 분해+이중검색 기여다.
- 단일홉: Baseline 대비 퇴행이 없어야 한다 (복잡한 파이프라인이 단순 질문을 해치지 않음을 증명).
- 오답을 고신뢰(confidence>0.8)로 내는 비율: Agent-basic 대비 절반 이하 (confidence 산식과 Agent-basic 산출법은 03에서 동일 정의를 먼저 확정해야 한다).

## 5. 산출 형식
`run_eval.py --mode {baseline|basic|ours_g|ours} --set {single|multi|typo}`를 실행하면 `results/{run_id}/metrics.json`과 `per_question.jsonl`이 생성된다.
리포트 생성기는 3단 비교표와 유형별 분해, 검사별 기여를 만들어 발표 표에 그대로 쓴다.
