# 07. 첫 번째 태스크 — W0.1 KorQuAD 수율 체크 (지금 바로)

## 왜 이게 1번인가
현재 데이터 전략에 **미검증 가정 2개**가 있고, 이 스크립트 하나가 둘 다 사실로 만든다:
1. "KorQuAD에서 브릿지 가능 쌍(answer(QA1)==entity(QA2))이 충분히 나온다" — 안 나오면 멀티홉 평가셋 전략 붕괴
2. "코퍼스를 평가 문항 원문서+이웃으로 역정의하면 정합이 보장된다" — 문서 목록이 여기서 나옴
부산물로 relation 분포가 나와 relations.yaml(W0.2)을 가설이 아닌 실측으로 확정한다.

## 스펙: scripts/w0_yield_check.py

### 입력
- KorQuAD 1.0 train+dev JSON (https://korquad.github.io/ 에서 다운로드, data/raw/에 배치)
- (선택) 한국어 위키 redirect 목록 — 없으면 문자열 매칭만으로 1차 수행

### 처리
1. 전 QA 로드 → (qid, question, answer, 문서제목, 문단) 평탄화
2. **정답 엔티티 정규화**: answer에서 조사 제거, 괄호 보조표기 분리 ("본(Bonn)" → {본, Bonn})
3. **브릿지 매칭**: answer(QA_a)가 question(QA_b)에 부분 문자열로 등장하는 (a,b) 쌍 탐색
   - a와 b의 원문 문서가 서로 달라야 함 (같은 문서면 멀티홉이 아님)
   - answer가 2글자 이하이거나 일반명사(불용어 목록)면 제외 (오매칭 방지)
   - 매칭 방식 태깅: exact / josa-stripped / alias
4. **relation 휴리스틱 태깅**: QA_a와 QA_b 질문 각각에 06 문서 relations.yaml 초안의 ko 표현 패턴 매칭 → 등장 relation 분포 집계 (미매칭은 "unknown"으로 집계 — unknown 상위 표현도 함께 출력해 새 relation 후보 발굴)
5. **코퍼스 문서 목록**: 브릿지 쌍 + 단일홉 평가 후보 문항들의 원문 문서 제목 집합 산출

### 출력
- `data/w0/bridge_candidates.jsonl` — {qa_a, qa_b, bridge_entity, match_type, relation_a, relation_b}
- `data/w0/relation_stats.json` — relation별 빈도 + unknown 상위 표현 30개
- `data/w0/corpus_doc_list.txt` — 문서 제목 목록
- `data/w0/report.md` — 아래 판정 기준 자동 채움

### 완료 기준 (report.md에 명시)
| 항목 | 판정 |
|---|---|
| 브릿지 쌍 수 ≥ 300 | PASS → 진행 / FAIL → KorQuAD 2.0 확장 or 위키 하이퍼링크 보조 결정 필요 |
| relation 상위 10종 커버리지 (전체 쌍 중 상위 10 relation으로 태깅되는 비율) | ≥60%면 스키마 10종 유지, 미만이면 스키마 교체 |
| 코퍼스 문서 수 | 하이퍼링크 이웃 포함 예상 문단 수가 3만 이내인지 추정 |

### 구현 메모
- LLM 사용 금지 (전부 문자열 처리 — 비용 0, 결과 결정적)
- 조사 제거는 간단 규칙으로 시작 (이/가/은/는/을/를/의/에서/으로/로 어미 strip), kiwipiepy 도입은 결과 부실할 때만
- 실행 시간 목표 < 5분, 진행률 출력
- 이 스크립트가 W1.6 build_multihop_set.py의 입력을 만든다 — jsonl 필드 누락 금지

## 이후 순서 (스크립트 결과 확인 후)
1. report.md 판정 확인 → relations.yaml 확정 (W0.2)
2. FAIL 시 인간에게 보고 후 대안 결정 (임의 진행 금지)
3. PASS 시 04_WBS.md의 W1부터 진행
