# 진행 기록 (실행 로그)

`handoff/`가 설계 계약(불변 스펙)이라면, 이 문서는 **실제로 무엇을 돌렸고 어떤 수치가 나왔는지**의 시간순 기록이다. 결정의 근거는 `handoff/02`(ADR)에, 수치·여정은 여기에 남긴다.

기간: W0~W1 (2026-07-22 ~ 07-24). 제공자: Upstage solar (임베딩 4096d + solar-pro), OpenAI 후순위.

**기록 규칙 (2026-07-24 00:33부터 적용)**: 이 시점부터 항목마다 시각(HH:MM)을 남긴다. 문제(버그·실패·
막힌 지점)는 "문제"로 별도 표시해 정상 진행과 구분한다. 커밋은 파일 단위로 쪼개고 Claude 서명을
빼기로 함(사용자 지시) — 아래 기록의 "커밋" 표기는 이 규칙 적용 이후 항목엔 해당 커밋 해시를 나열한다.

## W1.6 — 멀티홉 평가셋: KorQuAD 실패 → 2Wiki 전환
### 시도 A: KorQuAD 브릿지 자동 생성 (폐기)
- 브릿지 → LLM 패러프레이즈 → shortcut 필터로 gold 생성 시도.
- 결과: solar-pro·gpt-4o-mini **두 심판 모두 clean 사슬 ~3% 수율에 그마저 깨짐**(사람을 memberOf 대상으로, 날짜를 bornIn 답으로 등).
- 진단: KorQuAD 답↔질문 브릿지는 **어휘적 우연**이지 관계 사슬이 아님. 소스 한계(모델 문제 아님). W0.1 "1,969"는 허수.

### 조사: 기존 한국어 멀티홉 셋
- 없음. KoBLEX=법률, KorQuAD2=문서내 다중문단, AI Hub=뉴스 MRC → 부적합. 깨끗한 2-hop(2Wiki·HotpotQA)은 전부 영어.

### 시도 C: 2WikiMultihopQA 한국어 적응 (채택)
- 2Wiki는 (entity, relation, entity) 트리플 + 추론유형 내장 → **관계·타입 gold가 데이터에 있음**(LLM 추측 불필요). 라이선스도 파생 허용.
- 파이프라인: compositional 필터 → 관계를 우리 스키마로 매핑 → 엔티티 en→ko langlinks 정합 → ko 위키 intro를 코퍼스 증분 → solar-pro 질문 한국어화.
- **프로토타입 9/10 clean** (KorQuAD 0/10 대비). 확대 결과 **26문항** (관계매핑 330/850=39% → 한국어정합 → 채택 26).
- 산출: `data/eval/multihop_2wiki.jsonl` 26문항 + `data/corpus_2wiki.jsonl` 52문단.
- **한계: createdBy→{bornIn 15·nationality 6·diedIn 4·memberOf 1} 편향** (전부 "창작자→인물 속성" 단일 패턴). 다양성 위해 A(KorQuAD 정제, 병렬 진행)와 합산 필요.

### 통합
- `corpus.jsonl` + `corpus_2wiki.jsonl` = 10,667문단 재인덱싱(증분 임베딩 52개, 기존 재사용).
- 이 인덱스에 멀티홉 Baseline 측정 → 위 표.

## W1.3 — 지식그래프 (트리플 추출) — 게이트 PASS
- solar-pro 제약 추출 → NetworkX. 초판 정밀도 **63%**(사전지식 오염: 서울-capitalOf-대한민국 등 참이지만 근거없음, ADR-1 경계지점).
- 수정: 트리플마다 **문단 substring인 근거 문장(evidence) 강제**(없으면 폐기) + 축약(...) 금지.
- 재추출(352문단): 339노드/294엣지, **정밀도 87%(26/30) ≥ 85% PASS**. 관계 분포 createdBy 154 편향(2wiki director 문서). ✗는 createdBy의 연주자/작곡가 뉘앙스.
- 안정화: LLM 타임아웃 60s(solar hang 실측 후) + 100문단 체크포인트.
- **발견 1 (표기 불일치)**: 그래프 노드명 ≠ eval 엔티티명. "존 레논"→"존 윈스턴 오노 레논", "토머스 제퍼슨"→"제퍼슨". substring은 비연속 토큰에서 실패, **토큰 부분집합** 링킹으로 일부 해소(build_alias, 존 레논 연결됨).
- **발견 2 (커버리지 sparse)**: 더 근본적으로 일부 eval 엔티티(켄 번스 등)는 그래프에 **엣지 자체가 없음** — 그라운딩 추출이 인용 가능한 문장 없으면 놓침(정밀도↔재현율 트레이드오프). → **이 eval에서 그래프 backlink 검증은 sparse**.
- **결론**: Ours의 이득은 이 2Wiki eval에서 주로 **분해 + 이중경로(벡터+그래프) 검색**에서 나오고, 그래프 검증(backlink)은 커버되는 엔티티에서만. 다음 세션: (a) 토큰/임베딩 링킹 강화, (b) 그래프 추출 재현율↑(전체 코퍼스 확대), (c) decompose+resolve_hop로 retrieval 이득부터 실증.

## W2 (E2E) — decompose·ground·resolve_hop·finalize·bootstrap 구현 + 실제 26문항 실행

correct.py·decompose.py는 병렬 워크플로(2에이전트)가 구현(실 API로 자체검증 완료). ground.py·
resolve_hop.py·finalize.py·bootstrap.py·apps/cli.py는 이번 세션에서 직접 구현 — backlink C1
로직처럼 미묘한 의미가 프롬프트만으로 에이전트에 새어나가면 위험한 부분이라 직접 맡음.

### 구현
- `ground.py`: 앵커(alias exact→임베딩 top-1) + relation(exact→스냅→LLM타이브레이크→포기), ADR-6.
- `resolve_hop.py`: [4]검색(경로A+B 병합)+[5]검증(quad 주입)+[6]재질의. **backlink는 origin(vector/
  graph) 무관 균일 계산** — graph 출신이라 자동 통과시키지 않는다(C1 우회 차단). demo에 정상 케이스와
  contradict 하드veto 케이스 둘 다 포함, 둘 다 통과.
- `finalize.py`: final_op(결정적)+Path-Check(제약BFS)+응답조립(템플릿, LLM 생략).
- `bootstrap.py`: build_pipeline(mode) 조립 루트. FULL/SIMPLE/agent_basic/ours_g 4모드.
- `eval/run_eval.py`: ours/agent_basic/ours_g 모드 추가(EM 기반 정답률 — 05 C3의 완전한
  Recall@k(검색시점 전체 합집합 재랭킹)는 resolve_hop이 검색시점 문단을 아직 노출 안 해서 후속 과제).

### 실행 중 발견해 고친 진짜 버그 4건 (실측 없이는 안 보였을 것들)
1. **desc_check가 문단ID 문자열을 임베딩**하고 있었다(문단 텍스트 아님) — 상시 실패 버그. `state.pid2text`로
   실제 텍스트를 확보하도록 수정.
2. **그라운딩 타이밍 구조 버그**: 원래 hops 전체를 실행 전에 일괄 그라운딩했는데, 참조 hop({n})은 그
   시점엔 start가 아직 템플릿 문자열이라 앵커를 조회할 수 없어 start_node가 영구 None이 됐다. 그라운딩을
   executor 콜백 안, hop 단위(치환 후)로 옮겨 해결. finalize의 path_check도 start_node 필드 의존을
   없애고 매번 alias로 직접 정규화하도록 수정.
3. **SIMPLE/agent_basic 폴백의 expected.type="TERM"**이 사실상 항상 틀리는 억지 타입이라 type_check가
   상시 실패였다 — ""(제약없음 sentinel)로 바꿔 그 경우 type_check를 None(중립)으로 처리.
4. **Upstage 임베딩 API가 빈/공백 문자열을 400으로 거부**(실측) — desc_check 등에서 빈 문자열이 만들어질
   수 있어 어댑터 단일 관문(OpenAIEmbedder.embed)에서 자리표시자로 치환.

### 실제 26문항 실행 결과 (2Wiki gold, mode=ours)
| | 값 |
|---|---|
| EM 정답률 | 7.7% (2/26) |
| status 분포 | UNVERIFIABLE 25 · ANSWERED 1 |
| 평균 confidence | 0.47 |

**이 숫자를 Baseline Recall@10(61.5%, 동일 26문항 재측정 재현 확인)과 직접 비교하면 안 된다** — 서로 다른
지표다. Baseline은 "정답 문단이 top-10에 있는가"(검색), Ours EM은 "최종 답 텍스트가 정확히 일치하는가"
(다중hop 추론 후). 오늘 밤의 목표는 이 최종 KPI가 아니라 **메커니즘이 실제로 도는지**였고, 그건 증명됐다
(직접 구성한 정상 hop으로 backlink=pass 확정 케이스, contradict 하드veto 케이스 둘 다 코드로 재현됨).

### 26문항 결과의 원인 분해 (셋 다 진단됨, 신비 아님)
1. **그래프 커버리지 희박** — 트리플 추출이 552/10,667문단(~5%)만 처리했다(속도·비용 트레이드오프,
   이미 로그됨). `케네스 브래나`처럼 그래프에 노드는 있어도 outgoing edge가 0개인 경우가 있다 →
   graph 경로(backlink 신호)가 아예 없어 벡터 단독(약화모드)에 의존.
2. **decompose의 질문별 비결정성** — solar-pro는 temperature=0이어도 완전 결정적이지 않다. 같은
   질문("존 레논의 곡 '하느님'을 부른 가수는...")이 실행마다 다른 hop 구조를 냄: 한 번은 정상(존레논
   앵커→bornIn), 한 번은 의미상 난센스("존 레논을 만든 사람"=존레논의 부모를 찾는 hop을 만듦). 후자는
   **앵커가 질문에 이미 이름으로 노출된 경우** LLM이 불필요한 hop을 지어내는 프롬프트 취약점이다.
   부분적으로 원인 제공: `build_2wiki_ko.py`의 한국어화 프롬프트가 브릿지 엔티티를 질문에서 충분히
   가리지 못해("...가수는" 같은 자기지시적 문구), 일부 gold 문항 자체가 부자연스럽다 — 이건 평가셋
   품질 이슈지 파이프라인 버그가 아니다.
3. **검증 임계값(pass_ratio 0.75) vs 위 노이즈** — 정답을 후보로 찾고도(예: '리버풀'이 그래프
   backlink=pass로 최고점 후보) type_check가 그래프 타입충돌(~12%, 기존 로그) 때문에 실패해 3/4턱을
   못 넘는 사례를 직접 확인. **과신(틀린 답을 확신)이 아니라 과소확신(맞는 답도 검증 보류) 패턴** —
   신뢰성 관점에선 안전한 실패 방향이지만, KPI상 ANSWERED 비율을 깎는다.

## W1.3 확대 — 전체 코퍼스 그래프 추출 + 심각한 버그 발견·수정

W2 다음 단계로 "그래프 확대 → decompose 보정 → Agent-basic/Ours-G" 순서 지시받아 진행.

### decompose 프롬프트 보정 (규칙 9)
어젯밤 발견한 비결정성 실패(같은 질문이 실행마다 다른 hop을 냄 — "존 레논을 만든 사람" 같은 난센스
hop을 지어내는 경우)에 대응해, "질문에 이미 이름으로 등장하는 엔티티는 그 자체를 hop.start로 바로

### 4-arm 비교 (동일 26문항, 동일 그래프·decompose, EM 채점은 조사정규화 적용 후)

|---|---|---|---|---|

### 정직한 해석
- **Ours가 Agent-basic을 EM으로 이기지 못했다** — 11.5% 동률(3/26). N=26에서 1문항 차이가
  3.8%p를 흔드는 규모라 이 차이는 노이즈 안이다. "Ours > Agent-basic" 게이트(Day 8, 협상불가)는
  아직 충족도 반증도 안 됐다 — **이 수치로 통과 선언은 하지 않는다.**
- **분해만 추가하고 검증을 약화하면(Ours−G) 오히려 떨어진다**(7.7% < Agent-basic 11.5%) — 흥미로운
  신호: 멀티홉 체인은 실패 지점이 늘어나는데(hop1 실패=전체 실패), 그래프 검증 없이는 그 리스크를
  상쇄 못 한다. Ours(quad)가 Ours−G보다 나은 것(11.5%>7.7%)은 검증이 그 리스크를 일부 회수한다는
  방향의 근거다 — 다만 역시 N=26이라 잠정적.
- **세 arm 다 "ANSWERED"(확신) 표시했을 때 전부 오답이었다**(0/2, 0/3, 0/1). Ours가 확신 표시 자체를
  가장 적게 했다(1건, 세 arm 중 최소) — 과신 억제라는 설계의도 방향과는 맞지만, 표본이 너무 작아
  통계적 의미는 없다.
- **맞는 답 3개가 전부 Ours에서 UNVERIFIABLE로 분류됐다** — 어젯밤 발견한 "과소확신" 패턴이 그래프
  22배 확대 후에도 그대로 재현. 그래프 정밀도가 70%로 떨어진 것과 무관하지 않을 수 있다(가설,
  미검증): 노이즈 있는 backlink/type 신호가 pass_ratio 임계값을 걸리게 만드는 메커니즘은 어젯밤과
  동일.

#### 실패 패턴 1: 서술구를 엔티티명으로 추출
head/tail이 고유명사가 아니라 설명 문구 그 자체다. 규칙("고유명사 엔티티, 대명사·일반명사 금지")이

- "제이슨 마**의 데뷔 싱글** "Love"" -createdBy-> "마스" (head에 설명 접두, tail 엔티티명도 깨짐)

#### 실패 패턴 2: 관계를 "관련있음" 대용으로 오적용
memberOf(사람→단체)가 그 의미에 안 맞는 쌍에 붙었다 — 진짜 소속 관계가 아니라 "어떻게든 관련은
있다"는 느낌으로 라벨링됐다.

  스키마 방향 검사를 통과함)

#### 실패 패턴 3: 근거는 진짜인데 관계 라벨링이 틀림 (가장 어려움)
문단 인용은 정확한데(그라운딩 자체는 성공) 사실관계 해석이 틀렸다 — 근거문장 강제만으로는 못 잡는

분해·그래프·검증 전혀 없이 임베딩 검색→LLM 답변만 하는 최소 구현을 별도로 만들었다(사용자 요청,
나중에 대시보드에서 Ours와 나란히 비교용). `src/verihop/plain_rag.py` — bootstrap 어댑터 재사용(같은
코퍼스·모델, 공정 비교), `apps/plain_rag_cli.py`로 바로 테스트 가능. 실제 질문("존 레논은 어디에서
태어났나요?")에 "리버풀"로 정답. usecases/ 밖에 의도적으로 둠 — 파이프라인 계약을 우회하는 대조군이라
시스템의 일부가 아니다(check_layers.sh 대상 아님).

### 4-arm 재실행 (98% 정밀도 그래프) — EM은 그대로, 고신뢰오답에서 갈렸다

정밀도 70%→98% 그래프로 26문항 4-arm(Agent-basic/Ours-G/Ours)을 다시 돌렸다. Baseline은 그래프와
무관해 재실행 안 함(61.5% 그대로).

| Arm | EM (전→후) | ANSWERED 건수 (전→후) | 고신뢰오답 (전→후) |
|---|---|---|---|
| Agent-basic | 11.5%→11.5% (불변) | 2→3 | 2→3 (↑ 악화) |
| Ours−G | 7.7%→7.7% (불변) | 3→4 | 3→4 (↑ 악화) |
| **Ours** | 11.5%→11.5% (불변) | 1→0 | **1→0 (↓ 개선)** |

("고신뢰오답" = status가 ANSWERED인데 실제로는 틀린 경우. 05_평가_명세.md §4의 KPI "오답을 고신뢰로
내는 비율: Agent-basic 대비 절반 이하"를 이 지표로 직접 측정한 것.)

**해석**: EM은 세 arm 다 프루닝 전후로 완전히 동일했다 — 그래프 정밀도 개선이 "최종 답이 맞았는가"에는
전혀 영향을 안 줬다(솔직히 예상 밖이었다). 대신 갈린 건 확신 행동이다: Agent-basic·Ours−G는 그래프가
깨끗해지자 오히려 확신 표시(ANSWERED)가 늘었는데 여전히 전부 오답이었다(고신뢰오답 증가). **Ours만
반대로 줄어 0건이 됐다.** 즉 오늘 데이터가 뒷받침하는 주장은 "Ours가 더 잘 맞힌다"가 아니라 **"Ours는
확신 없인 답하지 않는다"** — 이게 05가 원래 정의한 KPI와 정확히 일치한다(고신뢰오답 0 ≤ Agent-basic의
절반). Day 8 게이트(EM 자체 비교)는 여전히 동률이라 미확정으로 남긴다. N=26이라 통계적으로 확정된
결론은 아니고, 방향성 증거로만 취급한다.

### (1) veto 시각적 확보 — 자연발생 안 함, 대신 실제 데이터로 메커니즘 직접 증명
모차르트 문서를 확인하니 "1781년 비엔나를 방문한 모차르트는... 비엔나에 머물며 명성을 얻었으나..."처럼
빈(비엔나) 언급이 실제로 많다 — 그래서 벡터검색이 빈을 후보로 헷갈릴 것으로 기대했다. 베토벤 문서도
같은 패턴("본에서 태어났으며... 빈에서 살았다"). 실제 질문 3회(모차르트 1회, 베토벤 1회, 존 레논
2회) 실행했지만 **전부 후보 1개(정답)로만 수렴** — extraction 프롬프트가 "태어났다"류 명확한 근거만
후보로 뽑도록 이미 잘 설계돼 있어서, "머물며/방문" 같은 비-출생 서술은 정확히 걸러냈다(정확도 관점에선
좋은 신호, 시연 모먼트로는 아쉬움).

억지로 자연발생을 더 쫓기보다, **실제 그래프 데이터로 검증 로직만 직접 호출**해 메커니즘을 증명했다:
```python
_backlink_state({'start_node':'모차르트','relation':'bornIn','direction':'fwd'}, '빈', G)      # → contradict
_backlink_state({'start_node':'모차르트','relation':'bornIn','direction':'fwd'}, '잘츠부르크', G) # → pass
```
가짜 그래프가 아니라 오늘 밤 만든 진짜 그래프·진짜 엣지로 확인됨. Streamlit 데모에서 "메커니즘 증명"
패널로 쓸 수 있다(라이브 E2E 우연 발생과는 별개로 명시).

부수 발견: 모차르트 문서는 "비엔나", 그래프 노드는 "빈"으로 표기가 달라(같은 도시, 다른 외래어
표기) alias 연결이 안 돼 있었다. `build_alias.py`에 `MANUAL_ALIASES`(비엔나→빈) 추가.

### 01:00 — [문제] 2차 재실행(b75q187ar)도 실패 — KeyError: 'pages'

### 01:50 — [문제] decompose SIMPLE 폴백 92.9% 발견·근본원인 3건 규명
mh2w_0010("영화 블러드 파더의 감독은 어디에서 태어났나요?") 등에서 최종답이 hop2 속성이 아니라

   고유명사와 동일하게 alias에 섞여 있어 "그 감독이 태어난 곳" 같은 정상 definition마다 누출로
   오판. `bootstrap._leak_check_alias_names()`로 occupation tail 값만 제외한 leak-check 전용
   뷰를 만들어 해결(그래프·grounding용 alias 자체는 안 건드림). → SIMPLE 폴백 92.9%→36.2%.
2. **그래프 범주어 오염**: 코퍼런스 실패로 실제 엔티티 대신 일반 범주어(영화·사람·책·소설·그룹·
   밴드)가 노드로 잘못 들어간 사례 발견(예: "그룹의 멤버"→실제 그룹명 대신 "그룹" 그 자체가
   노드). GENERIC_TERMS 블록리스트 추가(PRONOUNS와 동일 패턴, build_graph.py+build_alias.py),
   기존 그래프에서 6개 노드 직접 제거 후 alias 재구축(8,123개).
3. **짧은 alias 부분일치**: occupation 제외 후에도 36.2% 남아 재추적 — "신"·"원"·"나" 같은

   완화(LLM이 anchor_meta.name에 따옴표·접두어를 섞어 내보내는 포맷 흔들림 방어) 추가.
   → SIMPLE 폴백 36.2%→**8.7%**(남은 건 decompose 자체의 LLM 비결정성으로 판단, 기존
